# /docs/plans/

Substantive plan documents for each phase (and any mid-phase chunk large enough to need its own plan). These are the design lineage of the program — kept forever, marked complete when done, never deleted.

## Why these live here

A plan generated by a local tool and saved to `~/.claude/plans/<name>.md` is invisible to:

- Future Claude Code sessions (no shared context).
- Collaborators (no shared filesystem).
- Future-you reviewing *why* a decision was made (no git history).

Putting plans in this directory is the rule (see CLAUDE.md § "Planning lineage"). If a planning tool produced a local artifact, copy it here and reference the in-repo path from then on.

## Layout

```
/docs/plans/
  README.md                            this file
  phase-<N>-<name>.md                  substantive plan for a phase
  <feature-or-chunk>.md                substantive plan for a mid-phase chunk that warrants its own
  session_prompts/
    README.md                          (optional) — the priming-prompt convention
    phase-<N>-<name>.md                short prompt that primes a fresh session for this plan
    *.done.md                          archived prompts whose phase has committed
```

The plan and its priming prompt share the same suffix so lineage is obvious by filename.

## Frontmatter

Every plan starts with:

```yaml
---
name: phase-4-repo-extractor
title: "Phase 4 — Repo extractor"
phase: 4
status: draft               # draft | active | done | superseded
created_at: 2026-06-15
updated_at: 2026-06-15
completed_at:               # ISO date when status flips to done
supersedes: []              # list of plan names this replaces
superseded_by:              # set if status becomes superseded
locked_decisions: []        # decisions that future phases must not relitigate
---
```

`locked_decisions` is the section future plans grep. When a phase commits, the prompt's "Locked decisions (do NOT relitigate)" list moves up into the plan's frontmatter, so the audit trail is queryable.

## Status transitions

| Transition          | Trigger                                  | What changes                                                                 |
| ------------------- | ---------------------------------------- | ---------------------------------------------------------------------------- |
| `draft` → `active`  | Phase work begins.                       | First task lands as a commit.                                                |
| `active` → `done`   | Final phase commit lands.                | `status: done`, `completed_at`, commit SHAs noted in body, `locked_decisions` finalised. The sibling session prompt gets renamed `<name>.done.md`. |
| `*` → `superseded`  | A later plan replaces this one.          | `superseded_by:` set; the new plan's `supersedes:` lists this one.           |

A `done` plan is **not** edited further. If the design fundamentally changes, write a new plan that `supersedes:` the old.

## What does *not* belong here

- Day-to-day notes, journals, scratchpads — those live in conversation context or a personal scratch file.
- Operator runbooks for running the program — those live in `/docs/runbooks/` (user-facing) or `/command-center/runbooks/` (operator-facing).
- Engagement-specific plans for consulting clients — those live in `/consulting/engagements/<year>-<client>/`.
- Generated planning artifacts from local tooling, *unless* they've been promoted to a canonical in-repo plan.

## Reading order for a fresh session

1. CLAUDE.md (operating brief).
2. The relevant plan in this directory (e.g., `phase-4-repo-extractor.md`).
3. Whatever conventions and code the plan tells you to read.

The session prompt in `session_prompts/` exists to sequence those three reads.
